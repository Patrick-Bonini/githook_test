To mitigate downtime and data loss that results when disasters occur, offers replication as the solution for disaster recovery for deployments of s. When you use replication, you create identical in separate cloud regions or data centers (at least 50-100 miles away from each other). For simplicity, the main location where you use your s is referred to as the Primary site; the replicated site used as the alternate location is referred to as the Backup site. Each combination of Primary and Backup sites is referred to as a replication pair.

You can configure replication with any deployment ownership model (, , and ). You can also choose to use a mix of deployment ownership models when you configure your Primary and Backup sites. For example, you can choose to use a for your Primary site and a for your Backup site. The site where the Message VPN of the has a replication status of active is referred to as the Active site, while the site that is in a standby status is referred to as the Standby site. Client applications connect to the site using a custom hostname (sometimes referred to as an alternate hostname), which is assigned only to the Active site. The Active site can be the Primary site or Backup site depending on the replication status.

Limitations of Using Replication

The following are the limitations of using replication for s.

  • Only Enterprise services with high-availability (broker version 10.0 or later) are supported. Standalone and services are not supported.

  • Only s that use high-availability are supported. services are not supported.

  • Only an Active-Standby configuration is supported for replication; an Active-Active configuration is not supported.

  • s for both the Primary and Backup sites must be deployed in a Kubernetes-based region. VM-based deployments and regions do not support replication.

  • The Active site must be configured with a custom hostname. recommends that customers use only one hostname.

    Client applications must only connect to the using the custom hostname configured for the replication pair. Client applications must not use the IP addresses or the default, generated names for the s because during failover, they won’t be able to connect to the correct site. In addition, client applications are not permitted to connect Standby sites.

  • During the configuration of replication, you must disable and remove the cluster name from the Backup site. This temporarily causes inconsistencies during the configuration process in and that status of the , but are resolved upon completion of the procedures in this guide.

  • If your is part of an event mesh that was created with , you cannot use  to configure the external DMR links for the replication site. As described in this guide, you must manually configure the Dynamic Message Routing (DMR) links between:

    • each DMR node in the DMR cluster using
    • each DMR node that requires an external DMR link using
    • between the replication mates

    For more information, see Configuring Replication with an Event Mesh.

    Though you can view your event mesh using there may be validation errors when you perform a Health Check. You won't be able to use Mesh Manager to further manage the event mesh.

  • The replication configuration information and statistics for replication are not monitored by for replication. See Monitoring Best Practices for options.

  • To set up a replication pair, the Message VPN names must match. You cannot change the name of the Message VPN after service creation in .

    If you do not set the matching Message VPN name for your replication pair, you must delete the service and recreate it. For more information about how to set the Message VPN name at service creation time, see Setting the Message VPN NameSetting the Message VPN Name.

  • Advanced configuration options performed in the using must be manually synchronized. In addition, some configuration is not synchronized between the replicated pairs because there are properties that are considered unique to the in .

Best Practices

The following are best practices when configuring your s for replication help ensure that you replication functions as you expect:

  • recommends that you synchronize all service configuration in the between the s before you enable replication. If replication is already enabled, ensure that you synchronize all the configuration configured in the before you make any changes in . The configuration must be manually synchronized between the s using Cluster Manager > Manage in the .

    If you are making multiple configuration changes, you can remove the replication feature, make and manually synchronize the configuration changes between the s, and then re-enabling replication feature.

  • Config-Sync synchronizes the configuration between the replication pairs, such as client profiles and client usernames between the Active and Standby sites. However, advanced configuration options performed in the  using must be manually synchronized. In addition, some configuration is not synchronized between the replicated pairs because they are properties that are considered unique to the . For more information, see Config-Sync ReferenceConfig-Sync Configuration.

    These configuration options must be manually performed on both the on the Active and Standby sites for:

    • port configuration (connection endpoints)

      We recommend that the same ports and port numbers be configured on both the Active and Standby sites.

    • message spool sizes

      Message spool sizes must be the same size on both in your replication pair (both the Active and Standby site). After you enable replication, you cannot resize the message spool on the Standby site.

    • log forwarding

    • LDAP authentication

      If replication is enabled, the LDAP profiles must be synchronized prior to using LDAP authentication. If you use an LDAP profile before it is synchronized on the Standby site, this may cause Config-Sync state to go down.

    • SEMP request over message bus
