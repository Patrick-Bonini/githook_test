s

For the simplest deployment, s are deployed within the customer's account in , using the customer's choice of cloud provider and region. In this scenario, the s are dedicated to the customer and not shared. Because maintains control over this deployment, there isn't any additional customer overhead when adding services or regions. This is the standard deployment option, and is described in Getting Started with [%=Product-Names.cloud_product_short%].

Use Cases

This ownership model is a good choice when:

  • Applications and clients can connect over public internet
  • There are cost-sensitive workloads
  • The fastest time to market is required
  • A development or proof-of-concept environment is needed

s

In this deployment configuration, s are deployed in a region owned by and dedicated to the customer. In this dedicated virtual network, deploys a to orchestrate s. The agent creates a secure connection back to the and relays user commands from the console to the s. maintains control of the address space and network security groups.

With deployed in a , installs all the components with the sizing required by the customer. In this scenario, maintains the infrastructure, but the customer may provide the VPC/VNet and subnet CIDR blocks. Depending on the connectivity model, the customer may be required to provide sufficient information to establish network peering between the and customer virtual networks.

Use Cases

This model is a good choice if:

  • Single-tenant VPC/VNet ( i.e., network traffic isolation) is important
  • Connectivity to on-premises applications and network is required
  • The customer prefers VPC/VNetÂ management and security concerns to be included in the SaaS offering

s

For this scenario, s are installed in a customer's on-premises or cloud-based Kubernetes cluster. uses Kubernetes StatefulSets to manage the deployment of s.

Use Cases

This type of deployment is a good choice for:

  • Hybrid (cloud and on-premises) event mesh use cases
  • Data movement between legacy and Kubernetes-based applications
  • Highly sensitive data that cannot be on the cloud
  • Situations where an on-premises deployment is a requirement for other reasons (e.g., all applications are on premises)
