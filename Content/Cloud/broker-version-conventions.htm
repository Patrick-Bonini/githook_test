Release and Versioning Scheme of s

We do not use semantic versioning for our releases of s. In , s use an A.B.C.D-E numbering scheme where the first four segments (A.B.C.D) align to the same release and version of the underlying Solace . Solace does not use semantic versioning for theirs.

The A.B portion of the numbering scheme specifies the release of an event broker, while the C.D portion specifies the version portion of the release. The -E segment specifies the revision of the in .

s prior to August 2023 only use the A.B.C.D versioning scheme.

The following describes the details of the segments of the numbering scheme for s:

  • A.B is the release portion of the Solace .

    A release consists of a major. minor identifier. An increment to A or B indicates a newer release and differentiates one release from another. An increment to A resets B to zero (0). Increments to the major or minor number in a release can contain new features, enhancements, and fixes (both critical and security-related).

  • C is the first part of the version portion of the release that specifies the type of release.

    There are two types of releases, which are Preview (0) or Production (1). The numbering resets to 0 for each release (when B changes). New features, enhancements, and security fixes can be introduced between Preview and Production releases.

  • D is the second part of the version portion of the versioning scheme that identifies the build number of the Solace . Updates to the build number indicate new security fixes, critical fixes, and occasionally new features. The build number resets when C changes.

  • E indicates the revision of a release and version of a in . 

    This number resets to zero (0) when B changes. A value of zero indicates that it is the first revision of a specific release (A.B). Increments to the number indicate a new revision of the release and version of a software event broker in .
